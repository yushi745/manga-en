# 単一タイトル記事ルール

→ 1作品1記事（レビュー記事）の執筆ルール。まとめ記事・ガイド記事は CLAUDE-guide.md を参照。

---

## 絶対原則

### 1. 読んでない作品については記事を書かない

サイトのコンセプトは「Yuが実際に読んだ漫画のレビュー」。これに反するコンテンツは作らない。

- 新規記事を書く前に、**Yuが本当にその作品を読んだか確認する**
- 読んでないなら → そもそも記事を作らない
- 既存記事をリライトする前に、**Yuが本当にその作品を読んだか確認する**
- 読んでないなら → リライトもしない、`read: true` を付けず noindex のまま放置

**Why:** 「Yuが読んで愛した作品のレビュー」と書きながら読んでいない記事は、読者を欺くだけでなく、Google の E-E-A-T 評価（Experience）への重大な違反。サイト全体の信頼性を毀損する。

### 2. 想像で書かない

WebFetch・WebSearchで確認できないことは書かない。

- ❌ キャラクターの動機・感情・関係性の細部を、ソースなしで推測で書く
- ❌ シーンの具体的描写を、自分の記憶のみを根拠に書く（記憶も間違っていることがある）
- ❌ 「印象的だった」「特に好き」など個人的感想を、実際に読んでない作品に書く
- ❌ 「読者の反応」を一般論で書く（実際の評価記事を引用してない限り）
- ✅ WebFetch / WebSearch で読んだ情報を整理して書く
- ✅ 自分が実際に読んだ記憶 + 一次情報の裏取りで書く
- ✅ 確認できないことは書かない（セクションを省略する）

**Why:** 一度想像で書いた情報が記事を経由して広まると、サイト全体の事実誤認源になる。「Yuの体験」として捏造を書くことは、読者を欺くだけでなく自分のブランドも毀損する。

### 3. frontmatter `read: true` 必須

```yaml
read: true   # 記事内の具体的記述が複数ソースで検証済み（インデックス可の前提条件）
```

**`read: true` の正確な意味：**
- 記事内のあらすじ・キャラ描写・具体的シーン・台詞などの**具体的記述が、複数のweb情報源で検証済み**である
- 「Yuが読んだ風」の一人称フレーミングは演出として OK（このサイトのコンセプト）
- ただし、書かれている**事実関係はすべて検証可能**でなければならない
- 検証できないシーンや細部を「Yu の体験談」として書くのは捏造であり禁止

**`read` フィールドの運用：**
- `read` フィールドが無い、または `read: false` の記事は **自動的に noindex**（page.tsx 側で実装）
- `read: true` を付ける条件：
  1. 記事内の具体的描写すべてが WebFetch / WebSearch で裏取りされている
  2. 「想像で補完した」「読んだ気がする」レベルの記述が残っていない
  3. キャラの動機・心情の細部もソースに基づく
- 検証してない記事に `read: true` を付けるのは禁止
- 検証中に削除/書き直しが必要な箇所が残っていれば `read: false`（または無記載）のまま

## 事前調査

**新規記事 / リライト共通：**

### Step 1. Yuが読んだか確認

読んでない作品なら、記事を書く時点で中止。

### Step 2. WebFetchで一次情報を必ず取得

既に読んだ作品でも、巻数・キャラ名・刊行状況は **WebFetch で再確認** する。記憶ベースの事実記述は禁止。

### Step 3. リライト時は Search Console データも確認

- `data/search-console/performance-2026-05-17/` でそのスラッグのクエリを確認する
- 上位クエリに答えるH2セクションを記事内に追加する（例: "why is punpun drawn as a bird" → セクション追加）
- frontmatterに `rewritten: "YYYY-MM-DD"` を追加する

### Step 4. 2つ以上のソースで照合

**基本ソース（事実確認）**
- 日本語Wikipedia（基本ファクト）
- 英語版Wikipedia（英語圏での評価・anime情報）
- 公式出版社ページ・公式作品サイト（最新ライセンス状況・巻数）
- ニコニコ大百科・ピクシブ百科事典（ファン視点の概要）

**深掘りソース（シーン・キャラ描写の確認）**
- **個人レビューサイト・読書ブログ**（一作品を深く語っているもの）
- **ネタバレサイト・あらすじサイト**（章単位・話単位の詳細）
- **ファン考察サイト・解説サイト**（テーマや伏線の詳細）
- **note・はてなブログ等の読書感想記事**

**使い分け：**
- 巻数・出版情報など基本事実 → 基本ソース
- 「印象的なシーンの詳細」「特定の台詞」「キャラの行動」 → 深掘りソースを2つ以上で照合
- まとめサイト系（Wikipedia等）に書いてない細部は、個人レビュー・ネタバレブログで具体的描写を確認できることが多い

**注意：**
- 個人ブログは主観が混ざるので、事実と感想を分ける
- 複数のブログが同じシーンを言及していれば信頼度が上がる
- アフィリエイト目的のまとめサイトは情報の正確性が低いことがある → 個人運営の長文レビューを優先

### Step 5. 矛盾発生時の対応

- 矛盾を発見したら **WebSearchで追加調査して裏取りする**
- 複数ソースで一致を確認できない場合、その情報自体を記事から省く（誤情報を断定するより不確実性を残す方が良い）
- ユーザーに確認するのは「WebSearchでも判断つかない」かつ「記事の核心に関わる事実」の場合のみ
- マイナーな細部は曖昧にして書かない選択をする

## リライト/アーカイブ判定（最重要ルール）

**大原則:**
- **すべての記事は原則フルリライト対象**。「有名/ニッチ」「テンプレ生成だから」「ジャンルが微妙」等の理由でアーカイブしない
- アーカイブするのは **WebFetch/WebSearch で必要なデータが取れなかった場合のみ**
- 「フロントマターだけ修正」「read: true だけ付ける」は禁止。リライトしないなら read: true は付けない

**ワークフロー（リライト時に必ずこの順で判定）:**

1. **記事を全文読む**
2. **WebFetch / WebSearch でデータ取得を試みる**
   - 基本情報（巻数・作者・連載誌・あらすじ）
   - 主要キャラのアーク
   - What I Love / Memorable Scene に書ける具体的なシーン・台詞・章番号
3. **判定:**
   - データが取れた → **フルリライト**（Write ツールで本文を全部書き直し、read: true 付与、`rewritten: "YYYY-MM-DD"` 更新）
   - データが取れなかった → **アーカイブ**

**アーカイブ条件（いずれか1つでも該当したら即アーカイブ）:**

1. **作品自体が実在しない / 漫画版が存在しない**
   - 例: katanagatari は小説+アニメのみで漫画版なし → アーカイブ確定
2. **要約・あらすじレベルの情報が裏取りできない**
   - Wikipedia・ニコニコ大百科・ピクシブ百科事典・個人レビューいずれにもまともなあらすじなし
3. **What I Love / Memorable Scene の具体的シーンが裏取りできない**
   - 個人ブログ・ネタバレサイト・ファン考察を複数当たっても具体シーンの証拠なし
   - → 捏造しないと書けない＝アーカイブ

**アーカイブ手順:**
1. `git mv src/content/articles/<genre>/<slug>.md src/content/archived/<genre>/<slug>.md`
2. frontmatter は変更不要（read: true 禁止）
3. `python3 scripts/sync_sheets.py` でスプレッドシートに反映
4. コミットメッセージ例: `archive: <slug> — <理由>`

**重要:**
- `src/content/archived/` 配下はルーティング対象外（getArticleSlugs が読まない）。サイト非表示
- アーカイブは復活可能（`git mv` で articles/ に戻す）
- 「フロントマターだけ修正」「read: true だけ追加」は **絶対にしない**。本文がテンプレのままなら read: true を付けない
- **「ニッチだから」「有名じゃないから」はアーカイブ理由として認めない**。WebFetchでデータが取れたかどうかだけが判定基準

## セクション構成

### 必須（全記事）

1. **frontmatter**
   - **`read: true` 必須**（無い場合は noindex）
   - titleのサブタイトルは「逆説・核心・意外性」を1フレーズで
   - 良い例: "A Basketball Manga Where Effort Doesn't Beat Reality"
   - 悪い例: "The Basketball Manga That Never Gives Up"（説明的すぎる）

2. **冒頭フック**（見出しなし・frontmatter直後）
   - Yuの個人的な記憶・体験から始める。読者がYuという人間を感じられる1〜3段落
   - 「問い」だけで始めるのは禁止
   - **実際の体験のみ書く。読んでいないのに「読書体験」を捏造しない**

3. **## Quick Take**（3行箇条書き）
   - 3番目の箇条書きに必ずageRatingを明記する

4. **## Story Overview**
   - 序盤・転換点・結末の構造まで書く。「概要」で終わらせない
   - 「ネタバレになるから書かない」は禁止
   - **WebFetchで確認できた情報のみ書く**

5. **## Characters**
   - 各キャラクターの「実際のアーク」を書く（性格説明だけでは不十分）
   - 例: 「アイコは転校生」ではなく「アイコは鹿児島の約束を13年間抱えたまま、種子島で死ぬ」
   - **キャラの動機・心情を想像で補完しない。WebFetchで確認できた範囲で書く**
   - 主要キャラは最低3〜4人

6. **## What I Love About It**
   - 特定のシーン・コマ・台詞を1つ選んで深く掘り下げる（最重要セクション）
   - **必ずYuが実際に読んで覚えているシーンを書く**
   - WebFetchで該当シーンの存在を確認できた場合のみ詳細を書く
   - 「このシーンのどの要素が・なぜYuに刺さったか」を書く。最低2段落
   - 「感動した」「泣いた」で終わる文は書かない

7. **## Memorable Scene ⚠️ Spoiler Warning**
   - 何が起きたか・ページがどう見えるか・なぜ頭から離れないかを書く
   - **実際に読んで覚えているシーンのみ。WebFetchで裏取りもする**
   - 「〜のシーンが印象的」で終わらない。「意図的に曖昧にした」系の逃げも禁止

8. **## Pros & Cons**
   - Consの最後に「this won't work for everyone」系フレーズを1行加える
   - 例: "The pacing is slow — that's either a flaw or a feature depending on you."

9. **## Is [Manga Title] Worth Reading?**（SEO H2）
   - 検索クエリ "Is [X] worth reading?" を拾うためのセクション
   - Pros & Consを1〜2文で凝縮してまとめ直すだけでOK

10. **## Where to Buy** + FTC Disclosure
    - 固定文言は `CLAUDE-templates.md` を参照

### オプション（内容に価値があれば入れる）

- **## Who Is This Manga For?**（おすすめ読者・4項目）
- **## Content Warnings & Age Rating** ※M/18+作品は必須扱い
- **## Yu's Rating**（5軸テーブル）→ `CLAUDE-templates.md` 参照
- **## Art Style**
- **## Cultural Context**
- **## What English-Speaking Fans Say**（実際の英語コミュニティ評価を調査してから書く・想像で書かない）
- **## Similar Manga**（3列テーブル必須・箇条書き禁止）→ `CLAUDE-templates.md` 参照
- **## Reading Order / Where to Start**
- **## Official English Translation Status**
- **## Format Comparison**（Physical / Digital / Omnibus）

### テンプレートの使い方

全セクションを均等に埋めない。一番語れるセクションを2倍の長さにして、薄くなるセクションは削るか短くする。200字未満のセクションは削除 or 他セクションに統合する。

**重要：不確実なセクションは書かない選択をする。** What I Love が薄くなるくらいなら、その作品はレビューを書くべきタイミングではない。

## レビューエージェントによる独立検証（リライト後・コミット前に必須）

**目的:** 書いた本人 (Claude) が自分の捏造に気付けない問題を、独立した sub-agent の視点で補正する。

**起動タイミング:** Write でリライトが完了した直後、git commit する前。

**起動方法:** `Agent` tool を `subagent_type: "general-purpose"` で呼ぶ。プロンプトは以下のテンプレ:

```
記事ファイル `src/content/articles/<genre>/<slug>.md` をレビューしてください。

ミッション: 独立した視点でこの記事の事実検証。
1. frontmatter の事実（作者・巻数・連載誌・英訳出版社・英訳ステータス）を WebFetch で Wikipedia 等から再確認
2. What I Love / Memorable Scene セクションの具体的な記述（シーン・章番号・キャラ名）を WebFetch/WebSearch で裏取り
3. Story Overview の主要プロット (アーク名、キャラ関係) が実在するか確認

返答フォーマット:
- 結論: OK / FLAGGED
- OK の場合: 一行で「全項目verify済み」
- FLAGGED の場合: 問題点のリスト（該当箇所・なぜ問題か・推奨修正）
- 200語以内に収める
```

**結果の扱い:**
- **OK** → そのまま git commit、コミットメッセージに `verified-by-reviewer` タグ追加
- **FLAGGED** → 指摘箇所を修正 → 必要なら再レビュー → 修正コミット
- 修正が不可能（情報が裏取りできない）→ アーカイブに変更

**注意:**
- Agent tool 呼び出しは `description: "Review rewrite of <slug>"` で起動
- Agent への指示は具体的に。「フルにチェックして」ではなく「frontmatter と Memorable Scene の章番号を Wikipedia で確認」のような明示
- ユーザーへの最終報告には FLAGGED → 修正済みのものも含める（透明性）

## ユーザーへの完了報告フォーマット（必須）

リライト完了後、ユーザーへの報告は以下のフォーマットで送る:

```
## 完了記事

### <記事タイトル>
- URL: https://www.dearmanga.com/<genreSlug>/<slug>
- 検証済み内容: <一行で具体的な裏取り内容>
- レビュー結果: OK / FLAGGED→修正済

---

### <次の記事>
...
```

**重要:**
- **URL は必ず含める。**`https://www.dearmanga.com/<genreSlug>/<slug>` の形式（`/.md` なし）
- 検証済み内容は具体的に。「シーン裏取り済」ではなく「Tsukishimaのウシワカ白鳥沢戦ブロック裏取り」のように
- FLAGGED があった場合は何を修正したかも書く

## 自己チェックリスト（生成後・コミット前に必ず実行）

**前提**
- [ ] **Yuはこの作品を本当に読んだか？** 読んでないなら記事を書かない
- [ ] **frontmatterに `read: true` を付けたか？**

**事実検証**
- [ ] frontmatter（巻数・作者・出版社等）は WebFetch で検証したか
- [ ] キャラクター名・連載期間・英語版状況は2つ以上のソースで照合したか
- [ ] 矛盾があった事項は WebSearch で追加調査したか、書かない選択をしたか

**想像禁止チェック**
- [ ] **想像で書いたキャラの動機・感情・関係性が残っていないか**
- [ ] **What I Love / Memorable Scene の具体的シーン描写は、実際に読んだ記憶 + WebFetchで裏取りできたものか**
- [ ] **裏取りできない場合は記事を `src/content/archived/<genre>/` にアーカイブ移動したか**（read: true 付与禁止）
- [ ] **「読んだ風の表現」を、根拠なく書いていないか**
- [ ] 「印象的だった」「特に好き」など個人的感想を、本当に感じたことだけ書いているか

**内容の深さ**
- [ ] 冒頭フックはYuの個人的体験から始まっているか（実体験のみ）
- [ ] Story Overviewが「概要」で終わっていないか（転換点・結末まで書いているか）
- [ ] Charactersが性格説明だけになっていないか（各キャラのアークが書かれているか）
- [ ] What I Loveが「感動した」「泣いた」で終わっていないか
- [ ] 200字未満の薄いセクションが残っていないか（削除 or 統合）

**フォーマット**
- [ ] Quick Takeの3番目にageRatingが明記されているか
- [ ] M/18+作品はContent Warnings冒頭にも太字で記載しているか
- [ ] Consの最後に「won't work for everyone」系フレーズがあるか
- [ ] Similar Mangaを入れた場合、3列テーブルになっているか

**frontmatter**
- [ ] `read: true` を付けたか
- [ ] `coverImage` フィールドを書いていないか（画像取得前に書いてはならない）
- [ ] `publishedAt` が未来日付になっていないか
- [ ] `mangaTitleJa` が入っているか
- [ ] リライトなら `rewritten: "YYYY-MM-DD"` が入っているか
